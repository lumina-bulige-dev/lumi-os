# L-DEFENSE_PROTOCOL_v1.0  
契約トラブル・自力救済への OS 防御プロトコル  

※本ドキュメントは A：CORE 専用。  
正本は lumi-os/ リポジトリ内の本ファイルに従う。  
B／C／D／E は参照のみを認める。

---

## 0. Purpose（目的）

このプロトコルは、次のような場面で  
人間が「壊されない」ための最小 OS を定義する：

- 契約不履行・自力救済・脅し・ハッタリが混ざった係争  
- 個人 vs 事業者（販売店・ローン会社・回収業者 等）  
- 警察／弁護士／行政がグレー対応・逃げの姿勢を見せる場面  

本プロトコルの目的は：

1. 短期：  
   その場の不安・圧力に押し切られて、  
   「理不尽な支払い・泣き寝入り・自己犠牲」に流れないこと。
2. 長期：  
   ケース全体をログ化し、  
   将来の LUMI ユーザーが「同じ罠に落ちない」ための  
   OS 財産として蓄積すること。

---

## 1. Scope & Position（範囲と位置づけ）

- クラス：A-CORE（行動 OS ＋ 法的リスク防御レイヤー）
- 対象：
  - 個人ユーザー（Founder を含む）
  - LUMI CORE がサポートする「意思決定」レイヤー全般
- 非対象：
  - 具体的な法解釈・結論の断定  
  - 犯罪行為・報復行為の正当化

本 OS は：

> 「壊されないための動き方」と「ログの残し方」の  
> 型（プロトコル）だけを提供する。

---

## 2. Core Principles（中核原則）

1. **戦わないのは弱さではなく、賢さである**  
   - その場で殴り返さないこと ≠ 敗北。  
   - 「壊されない場所に立ち続ける」こと自体が勝ち筋。

2. **負け筋：感情ドライブ ＋ ノーログ**  
   - キレて脅迫・暴言 → 即「一発退場」リスク。  
   - 口頭だけで済ませる → 証拠ゼロで相手の土俵に落ちる。

3. **勝ち筋：質問で拘束し、記録で固定する**  
   - 主張をぶつける前に「質問」として投げる。  
   - 会話・LINE・通話はすべてログ（テキスト／録音）化して保存。

4. **“外部の正義役”を前提にしない**  
   - 警察・弁護士・行政は「味方」ではなく「別プレイヤー」。  
   - 彼らが動かない前提で、  
     自分側の OS とログで生存ラインを作る。

5. **戦う場所を自分で選ぶ**  
   - LINE・電話の消耗戦には付き合わない。  
   - 裁判・書面・記録に乗せやすい形だけを整える。

---

## 3. 相手側の典型ムーブ（パターン化）

車＋ローン＋自力回収ケースから抽出される典型パターン：

1. 本丸（契約不履行・車検未実施）には答えない  
2. 「支払義務」の話だけを繰り返す  
3. 「弁護士経由でしか話さない」と条件を押し付ける  
4. 「警察に通知した」とだけ述べ、具体を言わない  
5. 裁判所手続を経ずにロック・引き上げ等を先に実行する  
6. 当事者が不安になるまで、説明・対話を避け続ける  

目的は一貫して：

> 不安と無知を利用して、  
> 「支払って追認」か「諦めて泣き寝入り」させること。

---

## 4. LUMI 側の OS 的カウンター

### 4.1 “四重拘束質問” で動きを縛る

以下の 4 要素をセットにした質問を多用する：

1. どこの誰に（場所・相手）  
2. いつ（日時）  
3. 何を（具体内容）  
4. その結果何が返ってきたか（許可・回答の有無）

**例：**

> 「どこの警察署に、いつ、どのような内容で通知し、  
>  その結果、どのような回答または許可を得ましたか？」

これにより相手は：

- 正直に答えれば、警察照会で事実が確認される  
- 曖昧に答えれば、「警察を盾にしたハッタリ」が濃厚になる  
- 無視すれば、「説明拒否」のログが積み上がる  

→ いずれにせよ、どの選択肢も後で相手側に不利な証拠として機能する。

---

### 4.2 「支払再開＝追認」にしない

- 本丸（契約不履行・説明不足）に何も手当てされないまま支払を再開すると、  
  - 「問題はなかった」  
  - 「全て了解済みだった」  
  と評価されるリスクがある。

**OSルール：**

> 本丸への回答・是正案が無い状態では、  
> 「支払う／支払わない」の最終判断を安易に下さない。

---

### 4.3 感情ではなく構造で記録する

ログに残すのは：

- 誰が（販売店／ローン会社／回収業者／警察）  
- どの肩書きで  
- どの行為を  
- どの順序で  
- 何と言いながら行ったか  

のみ。

「ムカつく／クズ」などの評価は OS 外に置く。  
OS に乗るのは **事実と順序だけ**。

## 5. L-DEFENSE_FLOW_v1.0（行動フロー）

1. **事案発生**  
   - 契約違反・自力救済・威圧的行動を検知。

2. **即時ログ化**  
   - 日時・場所・相手・内容をメモ。  
   - 以後の会話は録音 or テキスト起こし。

3. **初動質問（First Lock）**  
   - 「どの契約条項」「誰の義務」「どの行為」が問題かを  
     自分で文章化。  
   - 相手に“質問”として送る（LINE／メール／書面）。

4. **反応観察**  
   - 本丸に答えず、「支払」だけを要求してくるか  
   - 「弁護士を通せ」「警察に通知した」を盾にするか

5. **第二ロック（Multi Lock Question）**  
   - 警察通知：  
     署名／日時／内容／応答をセットで質問。  
   - 弁護士限定：  
     契約・社内規定・法令等、根拠条文を質問。  
   - 契約不履行：  
     なぜ説明・是正案を出していないのか質問。

6. **分岐**  
   - 弁護士が参戦 → ログ一式を渡せるように整理。  
   - 誰も乗ってこない → 自己訴訟を含めた民事戦略へ。  
   - 刑事の可能性あり → 事実関係を整理した上で改めて相談。

---

## 6. Implementation Notes（実装メモ）

- この種のログは、将来の Heiankyo / 行動信用レイヤーで  
  「逃げずに構造で戦った」記録として評価可能。  
- SNS・ブログでは、実名ではなく  
  「O県の中古車販売事例」といった抽象化で公開する。  
- OS の“強い部分”（拘束質問パターンなど）は  
  一部を非公開レイヤーに置くことも検討する。

---

## 7. Effect（効果）

- 「やられっぱなし前提」ではない OS が立ち上がる。  
- 外部プレイヤーが動かなくても、  
  個人が自分のログと OS で防御ラインを持てる。  
- 将来、LUMI ユーザーが類似事案に直面したときに、  
  「これはあの車の事件と同じ構造だ」と気付ける  
  ケーススタディ OS になる。
