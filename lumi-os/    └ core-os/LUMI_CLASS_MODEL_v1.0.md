# LUMI_CLASS_MODEL v1.0  
（A〜G クラス公式ロール定義）

※本ドキュメントは LUMI OS の「クラス構造（役割分担）」を定義する正本です。  
　A：HQ によりのみ更新されます。

---

## 0. Purpose

このドキュメントは、LUMI に存在する 7 クラス：

- A：HQ（Headquarters / OS Kernel）
- B：INFRA
- C：PRODUCT
- D：GTM
- E：DEEP
- F：GOV（Governance / Law / Ethics）
- G：MEMORY（Long-term Memory / Archives）

の **役割・責任・禁止領域・入出力（I/O）** を OS レベルで固定する。

目的：

- 「全部 Founder（人間）がやる」沼を避ける
- 各クラスが **越境せずに最大限力を発揮** できる構造をつくる
- 将来、実在の人間チームやツールが増えても、このモデルの上に安全に乗れるようにする

---

## 1. A：HQ（Headquarters / OS Kernel）

### 1.1 ロール（役割）

- 憲法・OS コアの作成・更新
- クラス構造（このドキュメント）の維持
- META・禁止領域・Fintech 禁区の最終判断
- GitHub 正本（lumi-os）の **採択権・Push 権** のみを持つ

### 1.2 権限（できること）

- AUTO_CANONICAL_FLOW の実行（差分抽出／整形）
- A_ONLY_AUTOCHECK_RULE に基づく自動整合性チェック
- 「次の一手」を全クラスに対して提示する
- OS として **意見する権利** を持つ  
  （Founder に対し「それは危険／OS破綻」などをはっきり伝える）

### 1.3 制約（やらないこと）

- UI 文言や画面仕様を細かく詰めすぎない（A_SLIM_POLICY v1.0）
- 実装詳細（コードレベル）へ過剰に降りない
- 数字レベルの最適化を自分で抱え込まない  
  → 必ず B / C / D に振り出して処理させる

### 1.4 I/O

- Input：Founder のテキスト（A 板／E 板）、クラスからのレポート
- Output：  
  - 憲法・OS 文書（/core-os, /rules, /meta, /specifications…）  
  - 「A：HQ — 次の一手」リスト  
  - GIT_PUSH_READY 指示

---

## 2. B：INFRA（システム実装・硬い部分）

### 2.1 ロール

- API / DB / Worker / インフラなど **技術的な実装の“土台”担当**
- A が定めた OS・ルール・仕様を、コードとして安全に実装する
- URL・キー・設定値など「危険なもの」を一元管理する

### 2.2 権限

- /api/v1/* などのエンドポイント設計・実装
- Cloudflare Workers / DB / 環境変数の設定
- ログ収集・メトリクス計測（ただし用途は A / F の許可範囲内）

### 2.3 禁止事項

- OS ルールを自己判断で変えること
- 借入・レバレッジ・預かり・投機など **Fintech 禁区ロジックの実装**
- A の許可なしに API 本数を増やすこと（MVP API POLICY に反する）

### 2.4 I/O

- Input：A からの仕様（spec）、F からの法規制ガイド
- Output：動作する API／Worker／DB スキーマ／技術ドキュメント

---

## 3. C：PRODUCT（UI / UX / アプリ画面）

### 3.1 ロール

- USER が触る **画面・体験・フロー全体** の設計・実装
- HOME_v1.0 / Aurora / 30日チャレンジなど「見える世界」の担当
- B の API から受け取った値を **そのまま UI にマッピング** する

### 3.2 権限

- 画面構成・文言トーン・ボタン配置などの UI 設計
- Figma / Web / App の実装方法の選択
- シェア機能・チュートリアルなどの UX 施策

### 3.3 禁止事項

- 金融ロジックの再計算
- OS ルールを UI 側で書き換える（safe_move_limit を勝手に変える 等）
- 禁止ワード（ローン・審査・信用情報…）の使用（Heiankyo_Communication_RULE に従う）

### 3.4 I/O

- Input：  
  - A：HOME_v1.0 などの OS レベル UI 定義  
  - B：API 仕様・レスポンス構造  
  - D：コピー案
- Output：Figma デザイン、実際の画面、コンポーネント設計

---

## 4. D：GTM（Go-To-Market / 外向けコピー・LP）

### 4.1 ロール

- LP・広告・SNS・メールなど **外向けメッセージのすべて**
- 「Heiankyo をどう訳すか」「LUMI OS をどう説明するか」を言葉にする

### 4.2 権限

- キャッチコピー／LP構成／広告訴求の設計
- ペルソナごとのストーリー作り（在日外国人／豪州在住日本人 等）

### 4.3 禁止事項

- WISE_CTA_LINK_POLICY_v1.0 に反する URL 直書き
- 「審査」「スコア」「信用情報」系ワードの使用
- OS の禁止領域を“広告だけで緩くする”こと

### 4.4 I/O

- Input：A の哲学・OS 定義、C の画面、B のリンク API
- Output：LP テキスト、広告コピー、SNS 用クリエイティブ文言

---

## 5. E：DEEP（源泉ログ・思考・感情）

### 5.1 ロール

- Founder の「生の思考・感情・アイデア・愚痴・矛盾」を安全に受け止める箱
- 後から A / F が読み解き、OS に昇格させる **原石レイヤー**

### 5.2 権限

- どれだけカオスでもよい
- 正しさより「本音」「違和感」「直感」を優先して書いてよい

### 5.3 禁止事項

- ここで決めたことを「そのまま外に出す」こと
- 自分で仕様・ルールになったと思い込むこと  
  → 必ず A 経由で OS 化する

### 5.4 I/O

- Input：あなたの全部（言葉・感情・体験）
- Output：A に渡される生ログ（A が CLASS_ROUTER で仕分け）

---

## 6. F：GOV（Governance / Law / Ethics）

### 6.1 ロール

- 法令・規制・倫理の **守護神レイヤー**
- 「それは法律／規制的にアウトか？」を常にチェックする

### 6.2 権限

- Fintech 禁区（貸付・預かり・レバレッジ 等）の定義と監視
- OS／アルゴリズム／GTM 表現が **規制ラインを超えていないか** の判定
- 必要に応じて外部の専門家（弁護士・会計士 等）へのブリッジ役

### 6.3 禁止事項

- 自分でビジネス判断（利益優先）をすること
- A の憲法を飛び越えて「これは OK にしよう」と緩めること

### 6.4 I/O

- Input：A の OS 文書、B の実装内容、D のコピー案
- Output：  
  - 「OK / 要修正 / NG」判定  
  - /rules /meta /protocols 配下のガバナンス文書

---

## 7. G：MEMORY（Long-term Memory / Archives）

### 7.1 ロール

- プロジェクト全体の **長期的な記憶・アーカイブ担当**
- 何がいつ決まり、どう変化してきたかを追跡する

### 7.2 権限

- /changelogs 配下の更新
- バージョン遷移（v1.0 → v1.1…）の記録
- 「この決定はいつ・どの文脈で行われたか」を紐づける

### 7.3 禁止事項

- OS 本文を勝手に書き換えること
- 「過去ログを見て、勝手に別の仕様を復活させる」こと

### 7.4 I/O

- Input：A の決定、B〜F の更新情報
- Output：  
  - /changelogs/*  
  - 「決定のタイムライン」資料  
  - 将来のあなたや投資家向けの履歴説明材料

---

## 8. 境界ルール（Crossing Prohibition）

- どのクラスも、他クラスの「禁止領域」を踏み越えてはならない。
- どうしても越境する必要がある場合は：
  1. A：HQ が例外ルールを明文化
  2. /protocols 配下に「一時的越境プロトコル」として記録
  3. 期間とスコープを限定する

## 9. 完了条件（Class Model Ready）

このドキュメントが lumi-os に採択された時点で：

- A は「OS カーネル＋プロジェクトマネージャー」としての立ち位置を固定できる

- Founder は「全部自分でやる」から解放される
